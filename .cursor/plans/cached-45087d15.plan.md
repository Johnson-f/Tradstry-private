# Implement JWKS verification with Redis cache for Supabase Auth

1) Add JWKS cache backed by Redis

- Create a module (e.g., [`backend/src/turso/jwks_cache.rs`](backend/src/turso/jwks_cache.rs)) that fetches JWKS from `SupabaseConfig.jwks_url` and stores keys in Redis with TTL (e.g., 10–15 minutes). Use local in-memory fallback only for a single request if Redis is temporarily unavailable.
- Provide helpers to get a decoding key by `kid`, refresh on cache-miss, and reuse the previous cached set when fetch fails.

2) Reintroduce JWT verification using JWKS

- Add `jsonwebtoken` (RS256) dependency in [`backend/Cargo.toml`](backend/Cargo.toml).
- Update [`backend/src/turso/auth.rs`](backend/src/turso/auth.rs) to decode/verify Supabase JWT locally using the Redis-backed JWKS cache (issuer/audience checks, exp validation, leeway). Map failures to existing `AuthError` variants.

3) Wire middleware to use Redis-cached verification

- In `jwt_validator` (and any other call sites) switch from per-request Supabase API validation to the JWKS-based validator; ensure Redis client is available via `AppState` so workers share cache across instances.
- Keep behavior otherwise the same (attach `SupabaseClaims` to request extensions).

4) Configuration & resilience

- Add env/constant for JWKS cache TTL and Redis key namespace; document in `backend/README.md`.
- Behavior: if JWKS fetch fails but Redis has a valid cached set (still within TTL), continue; if no cache exists or the token is invalid/expired, return an auth error and rely on the client to refresh via Supabase. Do not mint tokens server-side.

5) Tests / validation

- Add a unit/integration test with a sample JWKS + signed token to ensure verification works, cache hit/miss paths, and that invalid/expired tokens return auth errors (forcing client Supabase refresh).
